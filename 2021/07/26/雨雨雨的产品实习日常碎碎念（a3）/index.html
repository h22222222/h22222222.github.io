给PD讲技术模块终于开业了！

2021年7月25日 星期日
 

1、和主管汇报之后的收益

 
新人阶段小细节：
 

  • 不懂就问，要告诉师兄你不知道什么，写会议纪要就是一个很好的方式
  • 不要怕犯错，新人阶段不犯错，之后肯定会继续犯错
  • 要先了解我们有什么，怎么来的，之后有什么发展
  • 技术方面的问题，多参会，多写会议纪要，遇到不会的术语多问，整理一些技术文档
  • 会议纪要之类的文档要注意格式模板，邮件要写标题，消息要反馈
  • 从最底层的事情做不丢人，培养自己解决问题的能力，做救火队员也没事

 
新人三板斧：体感、用户反馈、数据分析
 
体感：可以用user story的形式，角色、场景、需求、功能，如果找不到用户，就自己来体验一遍。
 
用户反馈：持续跟进
1）咨询：用户不会，用户来问你，说明产品可能在用户体验上出现一些问题，并没有清晰告知用户使用说明书
2）报bug：用户使用之后发现了问题，那这些可能就是概率非常小的一些边缘异常，配合用户解决，在之后的产品设计中需要考虑到这些异常用例。
 
数据分析：找问题的关键。数据剖面、同一时段横向对比、纵向对比

 
沟通交流：
 

  • 要先对齐目标，为啥要做，而不是上来就做，堆细节，之后发现方向错了，这时候返工的成本非常大。
  • 和研发的沟通：生活中要给予支持、激励驱动
  • 让别人相信你的前提是，你有足够的能力，敢于承担责任

 
对产品的优化和推广：
 
找头部、IP，抓关键，如果发现客户都会出现/聚集在一个路径上，那就厚脸皮去那个路径上找客户，获得反馈，再不行就和客户一起吃住，用同理心感化。
 
时间管理的重要性：
 
gtd，要有to do list，要有优先级
处在不同生命周期阶段的产品：
0-1：为客户解决什么问题，产品规划搭平台，这个阶段事物较为单一
1-100：产品成熟了，如何将标品推广出去，触客、扩客，这个阶段主要面向交付，项目型居多，产品规划上的工作就少了，事物相对杂，所以就需要高效的时间管理。
 

2、给PD讲技术（1）

 
这一期我总结了一下这段时间一直在写的接口文档。虽然这种文档应该是PD讲需求提给开发同学之后由他们来写，但是作为PD梳理架构一种非常好的方式，也建议各位PD可以写一写接口文档，从而增加对于技术的理解。
 

2.1 REST API （表现层状态转化接口）

 
参考链接：https://www.jianshu.com/p/75389ea9a90b
 
rest api是前后端分离的最佳实践，是开发的一套标准或者规范，不是框架。我在这将其理解成服务端API，而把js api 理解成前端API。

rest api特点：
1、轻量，直接通过http，不需要额外的协议，通常有post更新新建资源/get获取资源/put更新资源/deletec删除资源操作。
2、面向资源，一目了然，具有自解释性
3、数据描述简单，一般通过json或者xml做数据通讯
4、表现层的主语是资源，一本书或者一个网址，可以用URI来作为每个资源或者地址独一无二的识别符
5、每一个URI代表一种资源，REST API搭建起客户端和服务器端的传递桥梁，客户端通过post/get/put/delete四种http动词对服务器资源进行操作，实现表现层状态转化。
 
服务端API接口文档的撰写
 
1.接口说明：功能介绍
2.适用场景：
3.适用限制
4.API服务地址
5.请求方式
6.请求参数（输入参数，表格形式表示清晰一些）
参数名 含义 说明 是否必填 缺省值
7.返回结果参数（出参）
参数名 含义 生成规则 类型 说明
8.请求示例
9.返回示例
json或者xml数据格式
 

2.2 前端 JS API接口

 
JS API：
利用Javascript编写的接口，使用HTML5（前端展示）就能调用这些接口实现一些功能。
JS API主要用在一些简单的上图或者渲染上，数据量不能太大，不能包含具体的计算，所以使用范围比较局限，且通常需要和SaaS平台一起配套使用，展示接口能力。
 
前端API撰写：
在写JS API的时候要注意告诉前端，我们需要这个接口实现在页面上什么样的交互效果，一般前端都会有一些已有的接口可以调用，可以不用写入参和出参这些。
